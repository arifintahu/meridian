## 2. Diagnosis

The strategy is fundamentally fee-positive (92 wins, $36.9 fees), but the entire
net loss is 8 stop-loss closes that all show `range_efficiency = 100` — the position
never left the range, the token simply bled down while we held single-sided
`bid_ask` liquidity below the active bin (maximum token-down exposure). 7 of the 8
held >30 min in-range while bleeding, so they should have been culled by the
in-range drawdown exit (`-10%` / `30m`), but instead rode past it to −12…−22% and
fired the stop-loss — and the stop itself overshot its −12% trigger (realized
−14.8% to −22.2%) on gap-downs. This is a management/exit problem, not a selection
problem: no screening signal separates winners from losers, so screening thresholds
were left untouched. The in-range drawdown rule under-firing is also worth a
separate code look (possible drawdown-clock reset on minor in-range bounces above
the threshold).

Expected effect: catching the 7 slow bleeds near −8% instead of −14.5% saves ≈$3/position
(~$20–24), moving the run from −$36 toward break-even/positive. The 6-minute GOKHSHTEIN
gap-down remains uncatchable by config. Trade-off: more aggressive exits risk culling
positions that recover — but none of these 8 recovered, over a solid 134-close sample.

**Not changed (mixed/weak evidence):** all screening thresholds (no signal separated
winners from losers); volatility filter (mid-vol weakest but no clean cut-point);
LOW_YIELD / OOR rules (near break-even, not loss drivers).

## 4. Code changes (commit `—`)

**None — read-only evaluation.**

Follow-up investigation (post-eval): traced *why* the in-range drawdown rule fired
**0×** across all 134 closes. The recovery-grace clock fix (`2331f63`,
2026-06-24) was already active during this run, so a bounce-reset bug is **not** the
cause. The real cause is parametric, not a bug:

- Exit precedence in `state.js#updatePnlAndCheckExits` checks **stop-loss first**
  (line 570, returns) before the in-range drawdown rule (line 580).
- During the run the drawdown floor (`-10%`) sat only **2% above** the stop (`-12%`),
  and firing required a position to sit `≤ -10%` but above `-12%` for a continuous
  **30 min**. On volatile single-sided memecoins the `-10% → -12%` leg completes in
  far less than 30 min, so stop-loss always won the race — the soft rule was
  structurally starved.

The §3 config change is the correct fix for this: the new floor `-8%` sits **above**
the new stop `-10%`, giving the soft rule a real firing zone (cut at −8% after the
halved 15-min wait, before the −10% stop). No code change needed. Fast drops that
cross −8% → −10% in under 15 min still take the stop, but the tail is now capped 2%
tighter (−10% vs −12%).
