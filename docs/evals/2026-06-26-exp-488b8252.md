## 2. Diagnosis

**Tail-driven, and the cause is detection cadence — not threshold values.** All five loss events are *fast adverse moves that overshoot their exit rule between management ticks*: the four stop losses crossed −10%→−12% within roughly one 10-min management cycle (overshoot avg ~1.9%), and the trailing-TP fell 7.71% from peak before it was detected (target drop 1.5%). The loss concentrates by pool only because those pools happened to host the fast reversals (GLUE −$6.71 / Merlin −$5.69 / AIAIAI −$5.39), while WEN (+$8.21) and -USDC (+$4.09) repeat-deployed profitably — repeat-deploy count is not a clean lever.

**Answers to `exp-1f710961`'s re-check questions:**
1. **Does IN_RANGE_DRAWDOWN now fire on in-range bleeds?** → **No — 0× again** (now 0× across ~240 closes / 3 runs). The §4 grace-tolerance fix is sound; there were simply no *slow* bleeds to catch. Every in-range loss this run crossed the −10%→−12% band faster than the 30-min soft-stop dwell, so the hard stop always fired first. A dwell-based soft-stop is structurally mismatched to bleeds this fast.
2. **Did −12 cause false stop-outs or turn PnL positive?** → **No false stop-outs visible, and the change worked**: realized stop severity fell from **−17.30%** (prior run) to **−13.90%** here. Net stayed ~−$4 only because this run had **4 stop events vs 2** (market-driven), not because −12 misbehaved.

No screening change is defensible — third consecutive run with zero winner/loser separation.

## 4. Code changes (commit `c6e5593`)

Implemented the recommended fast-path stop loss this session.

**Finding during implementation:** the PnL poller was *already* detecting stop loss every 30s via `updatePnlAndCheckExits` (state.js:547) — so the original "extend the poller to check stop loss" framing was incomplete. The real defect was **two throttles on that detection**:
1. The shared `_pollTriggeredAt` cooldown (= `managementIntervalMin` = 10 min) gated *all* poller-triggered management, so a hard stop could be deferred up to a full management cycle if the poller had recently fired for a non-urgent exit (trailing/low-yield/OOR) on any position.
2. The per-position `break` meant a non-urgent exit on an earlier position masked a stop loss on a later one until the next tick.

### `index.js` — dedicated hard-stop-loss fast path in the PnL poller

Added a pre-scan at the top of the poller body (before the per-position loop) that triggers management the instant any position breaches `stopLossPct`, bypassing both throttles. Guard matches `updatePnlAndCheckExits`'s STOP_LOSS (skip suspicious PnL ticks). Tightened the poll interval 30s → 20s.

```diff
+      // ── Urgent: hard stop loss fires immediately ──────────────────
+      const slPct = config.management.stopLossPct;
+      const stopLossHit =
+        slPct != null
+          ? result.positions.find(
+              (p) => !p.pnl_pct_suspicious && p.pnl_pct != null && p.pnl_pct <= slPct,
+            )
+          : null;
+      if (stopLossHit) {
+        _pollTriggeredAt = Date.now();
+        log("state", `[PnL poll] STOP LOSS: ${stopLossHit.pair} ${stopLossHit.pnl_pct.toFixed(2)}% <= ${slPct}% — triggering management immediately`);
+        runManagementCycle({ silent: true }).catch((e) => log("cron_error", `Stop-loss poll management failed: ${e.message}`));
+        return;
+      }
-  }, 30_000);
+  }, 20_000);
```

Also updated the poller-cadence / cooldown references in `CLAUDE.md`. **Tests:** full unit suite green (63 pass / 0 fail); `node --check index.js` clean.

### `state.js` / `index.js` — skip the trailing-TP 15s recheck on severe drops (commit `2582234`)

Same latency family as the stop-loss cooldown. The trailing-TP exit detects a drop, then waits `TRAILING_DROP_CONFIRM_DELAY_MS` (15s) and rechecks before closing — whipsaw protection. But `resolvePendingTrailingDrop` records the **confirmation-time** PnL as the close reason, and this run's only trailing loss shows what that costs: AIAIAI peaked +4.13%, and the close reason was `current −3.58% (dropped 7.71%)` — i.e. during the 15s window the dump bled ~6% past the intended ~+2.6% trailing exit.

Added pure predicate `isSevereTrailingDrop` (state.js, 7 unit tests): a drop is severe — and skips the recheck — when gains are fully given back (`current ≤ 0`) or the drop is ≥ 2× the trailing target. The poller and the management cycle now confirm such drops synchronously (`confirmTrailingDropNow`) and close immediately, bypassing both the recheck and the poller cooldown. Marginal drops keep the 15s recheck, so the winning trailing exits (3× +1.84% in `exp-1f710961`, all small in-profit drops) are unaffected. **Tests:** 70 pass / 0 fail (also wired `in-range-drawdown` + `severe-trailing-drop` suites into `npm run test:unit`).
