## 2. Diagnosis

**Tail-risk, not selection — unchanged from `exp-5300018c`.** Signals barely separate
winners from losers (organic 78 vs 77, volatility 3.6 vs 3.4, fee/TVL 0.24 vs 0.20);
every loser passed screening cleanly, so no threshold would have caught them. The
structural failure mode is the single-sided `bid_ask` range below the active bin: on
upside breaks the position exits flat (the 26 "pumped" closes, +0.51% — harmless); on
**downside breaks it converts to the dumping base token and bleeds**. The decisive
detail this run: BRIM-SOL bled −18.7% while **in-range 98% of the time over 18 hours**
— an in-range bleed that the OOR-wait rule physically cannot catch. The only config
lever that reaches it is `stopLossPct`. What-if on this dataset: `-20` flags
bullieve+LIGMA (the two worst, ~$10 saved), `-15` additionally flags BRIM (higher
false-cut risk), `-25`…`-35` flag LIGMA only.

## 4. Code changes — new `IN_RANGE_DRAWDOWN` exit rule

The eval (§ 2) identified an in-range bleed that no exit rule catches: BRIM-SOL lost
−18.7% over 18h at 98% range efficiency. `OUT_OF_RANGE` never fires (it's in-range),
`STOP_LOSS` is deeper (−20%), `LOW_YIELD` passes (it earned fees), `TRAILING_TP` is
from-peak/upside. This change adds a **sustained-drawdown cutoff**: close a position
whose `pnl_pct` sits at/below a floor *while in range* for a wait window. Max-hold was
rejected — long holds are often winners (Merlin +3.05% at 429m), so the discriminator
is "down **and staying down**," not time held.

Config-gated under `management` (defaults below). `inRangeDrawdownExitEnabled` was set
`true` in `user-config.json` so it goes live on the runner at next `git pull`.

| Key | Default | Live value | Meaning |
|---|---|---|---|
| `inRangeDrawdownExitEnabled` | `false` | `true` | feature gate |
| `inRangeDrawdownPct` | `-10` | `-10` | PnL floor that starts the clock |
| `inRangeDrawdownWaitMinutes` | `60` | `60` | minutes PnL must stay ≤ floor before closing |

Verification: 10/10 new predicate unit tests, 55/55 full unit suite, syntax-clean; plus
10/10 isolated integration checks of the stateful path (clock set→accrue→fire,
recovery-clears, OOR-clears, disabled-noop, suspicious-tick-ignored, stop-loss-priority)
run against a backed-up `state.json` and restored byte-for-byte. Generated reason
reproduces the case: `In-range drawdown: PnL -18.70% <= -10% for 65m (limit: 60m)`.

### 4a. `state.js` — pure predicate + stateful clock + rule

New pure predicate (mirrors `isLowYieldClose`), exported for testing:

```js
export function isSustainedDrawdownClose({ pnlPct, drawdownExitPct, minutesInDrawdown, drawdownWaitMinutes, enabled }) {
  const wait = drawdownWaitMinutes ?? 60;
  return (
    enabled === true &&
    drawdownExitPct != null &&
    pnlPct != null &&
    pnlPct <= drawdownExitPct &&
    Number.isFinite(minutesInDrawdown) &&
    minutesInDrawdown >= wait
  );
}
```

Clock update inside `updatePnlAndCheckExits`, beside the OOR block (saved with the same
`if (changed) save(state)`). Only advances on trustworthy in-range ticks; an OOR or
suspicious tick clears/skips it:

```js
// Update in-range drawdown clock (sustained-loss timer).
if (mgmtConfig.inRangeDrawdownExitEnabled && mgmtConfig.inRangeDrawdownPct != null && !pnl_pct_suspicious) {
  const inDrawdown =
    currentPnlPct != null && in_range !== false && currentPnlPct <= mgmtConfig.inRangeDrawdownPct;
  if (inDrawdown && !pos.in_range_drawdown_since) {
    pos.in_range_drawdown_since = new Date().toISOString();
    changed = true;
  } else if (!inDrawdown && pos.in_range_drawdown_since) {
    pos.in_range_drawdown_since = null;
    changed = true;
  }
}
```

Rule check, immediately after the stop-loss return (softer, time-confirmed net below the
hard stop):

```js
// ── In-range sustained drawdown ──
if (!pnl_pct_suspicious && pos.in_range_drawdown_since) {
  const minutesInDrawdown = Math.floor((Date.now() - new Date(pos.in_range_drawdown_since).getTime()) / 60000);
  if (isSustainedDrawdownClose({
    pnlPct: currentPnlPct,
    drawdownExitPct: mgmtConfig.inRangeDrawdownPct,
    minutesInDrawdown,
    drawdownWaitMinutes: mgmtConfig.inRangeDrawdownWaitMinutes,
    enabled: mgmtConfig.inRangeDrawdownExitEnabled,
  })) {
    return {
      action: "IN_RANGE_DRAWDOWN",
      reason: `In-range drawdown: PnL ${currentPnlPct.toFixed(2)}% <= ${mgmtConfig.inRangeDrawdownPct}% for ${minutesInDrawdown}m (limit: ${mgmtConfig.inRangeDrawdownWaitMinutes ?? 60}m)`,
    };
  }
}
```

`trackPosition` initializes the new state field:

```diff
     out_of_range_since: null,
+    in_range_drawdown_since: null,
     last_claim_at: null,
```

No change to `getDeterministicCloseRule` — `updatePnlAndCheckExits` runs first in both the
management cycle (`index.js:257`) and the 30s PnL poller (`index.js:722`), and its exit
takes top priority, so the rule fires from one place and covers both.
