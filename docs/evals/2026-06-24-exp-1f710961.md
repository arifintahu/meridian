It fired **0├Ś in 81 closes ŌĆö 0├Ś across 134 closes over both runs** ŌĆö even after the prior eval halved its wait 60ŌåÆ30.

CONDOR is the smoking gun: it bled to ŌłÆ15.57% over **309 minutes**, spending an estimated ~100 minutes inside the ŌłÆ10%ŌåÆŌłÆ15% band (at its ~0.05%/min average rate). A 30-minute soft-stop should have fired easily. The only explanation consistent with that **and** "halving the wait did nothing" is the clock's design: `state.js` hard-reset `in_range_drawdown_since` on *any* tick back above the floor (old `state.js:472-473`), so a choppy bleed that pokes above ŌłÆ10% between management ticks kept restarting the timer ŌĆö "30 *continuous* minutes below the floor" almost never held. QUEST is the other mode: a fast dump (ŌłÆ19% in 89m) that crossed the ŌłÆ10%ŌåÆŌłÆ15% band in ~24 min (faster than the 30-min wait) and **overshot the ŌłÆ15% hard stop to ŌłÆ19.03%**.

**Answer to `exp-f55baea5`'s re-check:** (1) the ŌłÆ10%/30min rule still did **not** fire ŌĆö the blocker is structural (continuous-dwell reset), not the wait value, so halving the wait again would not help; (2) ŌłÆ15% produced no visible false stop-outs but also failed to cap the tail (let a ŌłÆ19% loss through). The reliable tail control is the hard stop, and it needed both a tighter cap and a working soft-stop ahead of it. No screening change is defensible (zero winner/loser separation; 4 losers).

## 4. Code changes (commit `2331f63`)

### `state.js` ŌĆö in-range drawdown clock tolerates brief recoveries

The soft-stop fired 0├Ś across two runs because the clock reset on any above-floor tick (see ┬¦2). Extracted the clock transition into a pure, tested `nextDrawdownClockState()` and made it grace-tolerant: a single above-floor tick opens a recovery window instead of resetting; only a recovery sustained Ōēź `IN_RANGE_DRAWDOWN_RECOVERY_GRACE_MIN` (15m, > one management cycle) clears the clock. Firing still requires the current tick at/below the floor, so a genuine bounce never forces an exit while a stale clock winds down.

New pure helper (covered by 8 added unit tests in `test/in-range-drawdown.test.js`):

```js
export function nextDrawdownClockState({ inDrawdown, since, recoverySince, now, graceMin }) {
  if (inDrawdown) return { since: since ?? now, recoverySince: null };
  if (since == null) return { since: null, recoverySince: null };
  if (recoverySince == null) return { since, recoverySince: now };
  const recoveredMin = Math.floor((now - recoverySince) / 60000);
  if (recoveredMin >= graceMin) return { since: null, recoverySince: null };
  return { since, recoverySince };
}
```

Also added: `in_range_drawdown_recovery_since: null` to the position init object, and the `IN_RANGE_DRAWDOWN_RECOVERY_GRACE_MIN = 15` constant. **Tests:** full suite green (63 pass / 0 fail; was 55, +8 new).
